Assignment Scenario 

Assignment Title:  Satellite-Ground Control Real-Time System Simulation in Rust 

Weightage: Simulation (40%), Individual Report (20%) 

Delivery Mode: Individual submission, within a collaborative simulation context 

Objective 

This  assignment  simulates  real-time  coordination  between  a  CubeSat  in  low  Earth  orbit  and  a  Ground  Control  Station 
(GCS).  Students  will  design  and  implement  one  component  of  this  system  using  Rust,  with  strict  real-time  constraints 
including deadlines, latency, jitter, safety, scheduling drift, and fault handling. 

Each student is assigned a role: 

•  Student A: Satellite Onboard Control System (OCS) 

•  Student B: Ground Control Station (GCS) 

Together,  both  components  form  a  distributed  system  where  correctness,  safety,  efficiency,  and  performance  must  be 
demonstrated. 

Learning Outcomes 

On completion of this assignment, students will be able to: 

1.  Discuss real-time systems, their constraints, and key attributes like concurrency and robustness. (LO1) 

2.  Design systematic and schedulable real-time systems for safety-critical contexts. (LO2) 

3.  Verify real-time scheduling techniques and evaluate system performance under load. (LO3) 

Assignment Components 

Student A: Satellite Onboard Control System (OCS) 

1. Sensor Data Acquisition and Prioritization 

•  Simulate three onboard sensors with distinct intervals and priorities. 

•  Ensure jitter is <1ms for critical sensors. 

•  Prioritize  sensor  data  in  a  bounded  buffer.  When  full,  simulate  data  loss  and  log  dropped  samples  with 

timestamps. 

•  Measure scheduling drift by comparing expected vs actual task start times. 

•  Track latency from sensor read to buffer insertion.

• 

If critical data (e.g. thermal) is missed for >3 consecutive cycles, raise a simulated safety alert. 

2. Real-Time Task Scheduling 

•  Schedule and prioritize tasks: data compression, health monitoring, and antenna alignment. 

• 

Implement scheduling using real-time strategies (e.g., Rate Monotonic or EDF). 

•  Support preemption: thermal control must override lower-priority tasks. 

•  Log all deadline violations (start delay or completion delay). 

•  Measure and report scheduling drift and task execution jitter. 

•  Log CPU utilization: % active time vs idle time. 

3. Downlink Data Management 

•  Compress and packetize data for transmission. 

•  Prepare data for downlink within 30ms of visibility window. 

•  Log transmission queue latency and buffer fill rates. 

• 

If downlink not initialized within 5ms, simulate missed communication. 

•  Trigger degraded mode if buffer exceeds 80%. 

4. Benchmarking and Fault Simulation 

• 

Inject faults (e.g., delayed or corrupted sensor data) every 60s. 

•  Log all faults and responses. 

•  Evaluate: jitter, drift, deadline adherence, fault recovery time, and CPU utilization. 

•  Recovery time must be <200ms, else simulate mission abort. 

Student B: Ground Control Station (GCS) 

1. Telemetry Reception and Decoding 

•  Receive telemetry via IPC or sockets, decode within 3ms. 

•  Log packet reception latency and reception drift (actual vs expected arrival). 

•  Trigger re-requests on missing/delayed packets. 

•  Simulate “loss of contact” if ≥3 packets fail in sequence. 

2. Command Uplink Scheduler 

•  Maintain a real-time command schedule. 

•  Enforce deadlines for urgent commands (≤2ms dispatch). 

•  Validate commands against system state using safety interlocks.

•  Log command deadline adherence and rejection reasons. 

3. Fault Management and Interlocks 

•  Simulate reception of fault messages from satellite. 

•  Block unsafe commands until resolved. 

•  Track interlock latency (detection to command block). 

•  Document and explain all command rejections. 

• 

If fault response time >100ms, trigger critical ground alert. 

4. System Performance Monitoring 

•  Benchmark uplink jitter, telemetry backlog, and task execution drift. 

•  Record missed deadlines, system load, and fault recovery metrics. 

•  All real-time actions must be logged with timestamps. 

Shared System Requirements 

All students must: 

•  Log scheduling drift: difference between scheduled and actual task start times. 

•  Track latency in all pipelines (sensor to buffer, packet to uplink, command to response). 

•  Record jitter: variation in periodic task timing. 

•  Handle simulated faults and log recovery time.
